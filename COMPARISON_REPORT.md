# Tesla-RAG 比較レポート（Cacel V1-V8 vs Roy V1-V7）

- Cacel: <https://github.com/cacel-man/Tesla-RAG>
- Roy: <https://github.com/HappyBirthdayRoy/Tesla-RAG>

## 目的

同一テーマ（Tesla IR RAG）での改善プロセスを比較し、次プロジェクトで再利用できる実装パターンを抽出する。

## 精度推移の比較

### Cacel（README公表値）
- V1: 10%
- V2: 30%
- V3: 60%
- V4: 80%
- V5: 80%
- V6-V8: プロダクト化（FastAPI / pytest / Docker）

### Roy（results実測値）
- V1: EM 0.20 / Contains 0.70
- V2: EM 0.00 / Contains 0.70
- V3: EM 0.10 / Contains 0.40
- V4: EM 0.10 / Contains 0.40
- V5: EM 0.00 / Contains 0.80
- V6: EM 0.80 / Contains 0.80
- V7: EM 1.00 / Contains 1.00

## アプローチ差分

- Cacelは初期から「ハイブリッド検索→テーブル対応→リランキング→CRAG」の王道構成を段階導入。
- Royは一度劣化（V3/V4）を経験し、失敗分析で「p4系の財務テーブル脱落」を特定。
- RoyはV5で検索前段（データ品質）を修正しContainsを回復、V6で回答正規化を入れてEMを回復、V7で残差問題を局所修正して100%に到達。

## 何が効いたか（共通）

1. 1バージョン1変更で原因帰属を明確化
2. 検索品質（Contains Gold）と回答品質（EM）を分離して追跡
3. データ品質（チャンク構造・テーブル保持）を先に改善
4. 検索改善後に回答整形でEMを取りに行く

## 失敗から得た教訓（Roy側）

- ハイブリッド検索は重み/候補設計を誤ると財務行を落として逆効果。
- EMは「正しい数値を含む文章」でも落ちるため、回答フォーマット制約が必須。
- 回答モデルの性能だけでは解決しない。前処理と評価設計が同じくらい重要。

## 次プロジェクトで再利用する実行テンプレ

1. Baseline（vector only）を固定して計測
2. Hybrid導入（BM25+vector）
3. テーブル/行構造を壊さないchunking
4. 必要ならrerank
5. 回答正規化（EM対策）
6. 残差問題を設問単位で局所修正

## 結論

- Cacel版は「再現性の高い王道改善ルート」を示した。
- Roy版は「劣化→原因特定→回復」のデバッグ駆動ルートを示した。
- 両者を統合すると、次プロジェクトの最短ルートは以下：
  - **データ品質を先に固める**
  - **検索の再現性を作る**
  - **最後に回答整形でEMを詰める**
